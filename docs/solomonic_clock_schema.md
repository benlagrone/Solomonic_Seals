# Solomonic Clock Unified Schema

Purpose: define a canonical data model so the clock UI, guidance engine, scripture lookup, election timing, Pericope integration, and Life OS layers all read from one structured source.

## Current State

The current `data/solomonic_clock_full.json` is generated by [src/generate_full_dataset.py](/Users/benjaminlagrone/Documents/projects/pericopeai.com/Solomonic_Seals/src/generate_full_dataset.py) and is organized around:

- `title`
- `generated_at`
- `visual_parameters`
- `layers.core`
- `layers.celestial`
- `layers.planetary`
- `layers.spirit`

That is good enough for the current visualizer, but it is not yet normalized for:

- runtime guidance resolution,
- scripture and pentacle joins,
- canonical hours,
- life-domain and virtue overlays,
- Pericope tool/context injection,
- future user-state and reflection tracking.

## Planetary Layer

Each planet should carry day, angelic, symbolic, and virtue metadata in one place.

```json
{

    "Mars": {
      "day": "Tuesday",
```

This should replace ad hoc joins between planetary rings, textual labels, and guidance hints.

## Zodiac Layer

The zodiac ring should be explicit rather than inferred from spirit sectors only.

```json
{
  "zodiac": [
    {
      "name": "Aries",
      "startDegree": 0,
      "endDegree": 30,
      "sectorStart": 0,
      "sectorEnd": 5
    }
  ]
}
```

Rule:

- every zodiac sign spans `30` degrees,
- every sign contains `6` five-degree sectors,
- `12 * 6 = 72` sectors.

## Wisdom Graph

Add a graph layer so the system can traverse relationships among sectors, virtues, domains, scripture, practices, planets, and shadows instead of relying only on one-step lookups.

Example structure:

```json
{
  "wisdomGraph": {
    "nodes": [
      { "id": "leraje", "type": "sector" },
      { "id": "temperance", "type": "virtue" },
      { "id": "relationships", "type": "domain" },
      { "id": "proverbs_15_1", "type": "scripture" },
      { "id": "pause_before_speaking", "type": "practice" }
    ],
    "edges": [
      { "from": "leraje", "to": "temperance", "relation": "corrected_by" },
      { "from": "temperance", "to": "relationships", "relation": "governs" },
      { "from": "temperance", "to": "proverbs_15_1", "relation": "illustrated_by" },
      { "from": "temperance", "to": "pause_before_speaking", "relation": "practice" }
    ]
  }
}
```

Recommended node types:

- `sector`
- `planet`
- `virtue`
- `domain`
- `scripture`
- `practice`
- `shadow`

Recommended relation types:

- `corrected_by`
- `governs`
- `illustrated_by`
- `practice`
- `countered_by`
- `aligned_with`

Graph-discipline rule:

- `virtue` is the primary hub node,
- attach `sector`, `scripture`, `practice`, `domain`, and mentor/author metadata to virtues rather than building arbitrary peer-to-peer links,
- avoid direct `author -> author`, `sector -> author`, and `scripture -> sector` edges in the canonical model unless there is a narrowly justified derived view,
- this keeps the graph bounded as authors, passages, and practices scale.

This can ship embedded in the normalized dataset or as a separate `data/wisdom_graph.json` file that the runtime loads alongside the canonical schema.

## Guidance Resolver

The guidance panel should become a thin resolver over structured data instead of an ad hoc formatter.

```javascript
function generateGuidance(state, data) {
  const sector = data.sectors[state.currentSector];
  const pentacle = data.pentacles[sector.pentacle];
  const virtue = resolveVirtue(state, data);

  return {
    virtue,
    guidance: sector.guidance,
    scripture: sector.scripture,
    pentacle
  };
}
```

This makes the UI, Pericope chat layer, and oracle generator all resolve from the same source.

The virtue resolver should merge signals from:

- sector,
- planet,
- canonical hour,
- weak or highlighted life domain,
- scripture virtue tags.

The rule generator should then read:

- resolved virtue,
- weakest or highlighted domain,
- canonical-hour practice type,
- current planetary and sector context,
- wisdom-graph path,
- wisdom-index scripture anchors,
- matching practice-library entries.

The wisdom index should then read:

- resolved virtue,
- selected scripture references,
- mentor registry authority tiers,
- current mentor mode,
- any Christ- or Gospel-priority anchors,
- practice candidates linked to the active virtue.

It should return:

- canonical scripture anchor first,
- mentor candidates ordered by authority and relevance,
- practice options that remain aligned with that anchor.

The narrative engine should then read:

- planet and canonical-hour templates,
- resolved virtue and practice,
- selected scripture anchor,
- current reading depth,
- any sector warning or emphasis.

The mentor layer should then read:

- resolved virtue,
- scripture anchor selected by the Wisdom Index,
- selected or rotated mentor,
- mentor candidates ranked by the Wisdom Index,
- mentor style and tone metadata,
- linked Pericope author sources.

The guidance resolver should also be able to delegate to a Pericope guidance API:

```text
POST /lifeos/guidance
```

Example request:

```json
{
  "planet": "Mars",
  "sector": "Leraje",
  "virtue": "Temperance",
  "domain": "Relationships"
}
```

Example response:

```json
{
  "scripture": "Proverbs 15:1",
  "mentor": "Solomon",
  "reflection": "A gentle answer preserves peace.",
  "practice": "Respond calmly in conflict."
}
```

## Migration Plan

Suggested migration from the current `layers` model:

1. Preserve the current generator as the source of truth for sector ordering and existing visual data.
2. Introduce a schema-normalization step that emits:
   - `metadata`
   - `planets`
   - `zodiac`
   - `sectors`
   - `pentacles`
   - `scripture`
   - `canonicalHours`
   - `lifeDomains`
   - `virtues`
   - `practiceLibrary`
   - `wisdomGraph`
   - `wisdomIndex`
   - `narrativeTemplates`
   - `mentors`
3. Update the visualizer and guidance engine to read through a single normalized access layer.
4. Keep compatibility shims for the old `layers.*` shape until `web/clock.js` and related scripts stop depending on it.
5. Add validation for:
   - 72 sector continuity,
   - 12 zodiac spans,
   - 6 sectors per zodiac sign,
   - sector-to-pentacle references,
   - scripture reference existence,
   - canonical-hour count,
   - life-domain count,
   - virtue reference validity across planets, sectors, scripture, and domains,
   - practice-library coverage for required virtues,
   - wisdom-graph edge validity and node existence,
   - wisdom-index virtue coverage and mentor/scripture reference validity,
   - Pericope chunk metadata completeness for author, source, virtue tags, and life-domain tags,
   - narrative-template coverage for planets, canonical hours, virtues, and practice types,
   - mentor registry completeness for id, virtue focus, sources, and style,
   - timeline-entry completeness for required history fields.

## Why This Matters

Without a unified schema, the system remains visually strong but logically fragmented. With the schema normalized:

- the clock UI becomes easier to reason about,
- guidance resolves from structured joins instead of scattered constants,
- Life OS layers can build on the same symbolic core,
- Pericope and oracle generation can consume the same runtime state,
- future reflection, scoring, and user-state systems have a stable foundation.
